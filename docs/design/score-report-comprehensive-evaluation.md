# 成绩报告 — 综合评价 设计文档

## 定位

**一个临床综合技能考官系统**。给定任意考核项目组合及其对话记录，还原学员的认知过程，对深层能力维度做出推断，给出有路径的发展建议。

核心方法论：**逆向设计**——先定义"我要知道什么"，再定义"要知道这些需要分析什么"，最后定义"需要哪些输入物"。输入物不可盲信，可能存在不完整或错误。

---

## 输出定义

报告回答四个层次的问题：

| 层次 | 读者问题 | 报告 Part |
|------|---------|-----------|
| 我是谁？ | 我是什么样的医生？我的发展阶段在哪里？ | **Part A** 能力发展阶段定位 |
| 我是怎么做的？ | 在刚才的诊疗中，我的大脑里发生了什么？ | **Part B** 认知过程还原 |
| 这揭示了什么？ | 这些表面行为背后，有什么统一的模式？ | **Part C** 整合推断 |
| 我该往哪走？ | 接下来优先改什么？怎么改？ | **Part D** 发展导航 |

---

## Part A：能力发展阶段定位

### 我要知道什么

| 输出项 | 说明 |
|--------|------|
| **发展阶段** | S1-S5 的哪个阶段？（各维度可不同） |
| **一句话画像** | 作为临床从业者最鲜明的特征概括 |
| **能力标签** | 2-4个关键词（强项+待发展+风险） |
| **维度可评估性** | 手头数据能判断哪些维度、不能判断哪些 |
| **维度不均衡提示** | 哪个维度落后/超前于整体 |

### 需要什么分析

**维度阶段判定**：对每个可评估维度，根据剖面证据判断其发展水平。

判断依据：
- S1（知识驱动）：按教科书清单逐项执行，没有明确的假设活动。提问是"覆盖式"而非"追踪式"
- S2（规则驱动）：能按流程完成，识别典型表现。有模糊的假设意识但不会主动验证
- S3（模式驱动）：能自动激活假设并定向追问。但假设一旦形成就收窄，忽略不一致信息。**这是多数人停留最久的阶段**
- S4（情境适应）：能根据情境灵活切换策略，直觉和分析流畅交替
- S5（专家）：信息不足时仍能做出安全决策，能处理复杂不确定性

**维度不均衡分析**：各维度阶段判定值的横向比较。最大差距≥2个阶段 → 显著不均衡，写入 Part A 提示。

**可评估性判定**：对照观测矩阵，盘点手头可用的考核项目类型 → 每个维度是否有足够的观测窗口。

### 需要什么输入

| 输入物 | 优先级 | 用途 |
|--------|:---:|------|
| Part B 各剖面报告 | **P0** | 各维度的行为证据来源 |
| Part C 维度综合判定 | **P0** | 各维度的最终评级和置信度 |
| 考生专业和培训阶段 | P3 | 阶段判定的外部锚点（如"S2对住培第二年来说偏低"） |

## Part B：认知过程还原

### 我要知道什么

对每个考核项目，回答五个递进问题：

| 层次 | 问题 | 产出 |
|------|------|------|
| **L1 信息覆盖** | 关键信息点覆盖了多少？遗漏了什么？ | 覆盖热力图 + 遗漏清单 |
| **L2 行为策略** | 怎么组织行为的？策略模式是什么？ | 策略类型判定 + 行为特征描述 |
| **L3 认知过程** | 大脑里发生了什么？假设怎么形成和演化的？ | 假设活动时间线 + 推理模式判定 |
| **L4 安全行为** | 红旗征象被关注了吗？与假设活动的关系？ | 安全覆盖清单 + 安全行为模式 |
| **L5 关系质量** | 患者在这个对话中是什么样的存在？ | 情感线索捕捉率 + 关系弧线 |

**五个层次不是五个独立事件，而是同一个诊疗过程的五个观察窗。** L1-L2 是表层行为，L3 是认知核心，L4-L5 是 L3 的行为后果。最关键的洞察往往在层次之间的关联中：比如假设收窄（L3）的精确时间点，恰好也是安全筛查停止（L4）和情感回应消失（L5）的时间点——这三个事件是同一认知行为的表现。

### 六个剖面类型

考核项目不同，分析焦点不同。系统支持六种剖面，每种有独立的分析指令：

| 剖面 | 分析焦点 | 最大深度在 |
|------|---------|-----------|
| 病史采集 | 假设驱动的追问策略 | L3 假设活动重构 |
| 体格检查 | 查体策略与病史假设的关系 | L2 行为策略 + L3 关联 |
| 人文沟通 | 共情的真实度和关系弧线 | L5 关系质量 |
| 精神检查 | MSE系统性和治疗性互动 | L2-L5 五个维度全部高频 |
| 病例分析 | 推理链完整度和鉴别广度 | L3 认知过程 |
| 病历书写 | 信息整合和结构化 | L2 行为策略 |

### 需要什么输入

| 输入物 | 优先级 | 用途 |
|--------|:---:|------|
| **对话记录** | **P0** | 认知过程还原的原始证据。所有五个分析层次都从这里提取 |
| **病例原文** | **P0** | "标准答案"参照：关键信息点清单、红旗征象清单、正确诊断和依据 |
| 评分表数据 | P2 | L1覆盖分析的辅助框架。不存在时，直接用病例原文做覆盖对照 |
| 考核项目类型标签 | P1 | 确定使用哪个剖面类型的分析指令。不存在时从对话内容推断 |

**评分表在 Part B 中的角色**：它是一个预定义的"检查单"，可以加速 L1 覆盖分析。但它不是必需的——即使没有评分表，系统可以直接用病例原文做覆盖对照（病例原文中包含的关键信息点 = 应该采集到的信息）。评分表存在时，做分数信度评估；不存在时，Part B 照常运行。

## Part C：整合推断

### 我要知道什么

| 输出项 | 说明 |
|--------|------|
| **维度综合判定** | 五个维度各自的综合评级，证据是否收敛，置信度 |
| **跨维度根因分析** | 多个表面问题是否指向同一个深层原因？ |
| **数据局限性声明** | 基于手头数据，什么能判断什么不能？ |

### 需要什么分析

#### 维度综合判定

对每个维度：
1. 收集所有剖面中该维度的证据片段
2. 判断证据是否收敛（多个剖面指向同一结论）还是发散（不同剖面呈现不同表现）
3. 收敛 → 置信度高；发散 → 本身即发现（可能需要报告"你的XX能力在不同情境中表现不一致"）
4. 单一来源 → 置信度标注为中等，承认局限性

#### 跨维度根因分析

当多个维度同时暴露问题时，追问：**它们是否指向同一个上游原因？**

这个分析的认知行为是：
- 识别跨维度的重复模式（如"收窄"出现在推理、安全、技能三个维度）
- 提出根因假设 → 主动寻找反例 → 排除替代解释
- 判断根因的干预优先级（是否涉及安全？）

排除对立假设是最关键的一步。面对同一组表面问题，可能有多重解释：
- "漏了红旗征象" → 是知识不足（不知道要问）？还是认知习惯（锁定了另一个假设后不再考虑）？还是态度问题（觉得不重要）？
- 不同的根因指向完全不同的干预方案

#### 数据局限性声明

逆向使用观测矩阵：我手头有剖面A和剖面B → 维度X有强证据，维度Y有中证据，维度Z无证据 → 分类声明。

### 需要什么输入

| 输入物 | 优先级 | 用途 |
|--------|:---:|------|
| **Part B 所有剖面报告** | **P0** | 所有维度证据的聚合来源 |
| 临床认知偏误知识 | 知识库 | 根因假设的生成和排除替代解释的参照 |
| 可用的剖面类型清单 | P1 | 数据局限性判定 |

**Part C 不接触原始对话记录**。它的输入是 Part B 的结构化剖面报告——这已经是对原始数据的压缩和提炼。

## Part D：发展导航

### 我要知道什么

| 输出项 | 说明 |
|--------|------|
| **优先级决策** | 先改什么？为什么它排第一？ |
| **核心处方** | 针对根因的最有效干预——当前状态→目标状态→方法→检验标准 |
| **次要建议** | 非根因但值得关注的问题 |
| **推荐资源** | 与问题直接匹配的学习材料 |

### 需要什么分析

**优先级决策逻辑**：
1. 涉及安全的根因 → 最高优先级
2. 是其他问题的上游根因 → 次高优先级
3. 孤立问题，不影响其他维度 → 次要建议

**核心处方设计**：
- 处方必须针对 Part C 识别的根因，而非针对表面问题
- 需要明确定义"当前状态"和"目标状态"——让学员知道从哪到哪
- 需要可执行的练习方法，而非笼统的"加强练习"

### 需要什么输入

| 输入物 | 优先级 | 用途 |
|--------|:---:|------|
| **Part C 根因分析** | **P0** | 优先级排序和处方目标的依据 |
| **Part B/C 中的其他发现** | P1 | 次要建议的来源 |
| 教学方法/认知训练知识 | 知识库 | 干预方法的匹配 |

## 分数信度评估

报告中的独立模块，不属于四Part，但对读者理解分数的可信度至关重要。

### 我要知道什么

- 评分表给出的分数准确吗？高估了还是低估了？
- 如果存在偏差，哪些评分项出现了偏差？原因是什么？

### 需要什么分析

**校准逻辑**：将评分表的"覆盖判断"与 Part B 的"质量判断"逐项对比。

评分表的每一项记录的是"做了没有"（存在性判断）。Part B 的剖面报告记录的是"做得怎么样"（质量判断）。**当"做了"但"做得不好"时，评分表存在高估。当"没做"但"临床上不需要做"时，评分表存在低估。**

偏差方向判定：
- 高估（评分表 > 实际）：学员按清单覆盖了项目但执行质量浅。常见于"沟通技巧"类项目
- 低估（评分表 < 实际）：学员有意取舍，遗漏了低价值项目但聚焦了高价值项目
- 准确：评分表反映的水平与对话记录中观察到的质量一致

### 需要什么输入

| 输入物 | 优先级 | 用途 |
|--------|:---:|------|
| **Part B 剖面报告** | **P0** | 质量判断的来源 |
| **评分表明细** | **P0** | 逐项的"已做"标记和分数 |

**评分表仅在分数信度评估中是 P0 输入物。** 在报告的其他部分，它是 P2（辅助框架）。

---

## 评分表匹配度评估

报告中的独立模块。一个整体性判断，用于校准读者对分数的信任程度，而非对评分表逐项审计。

判断的是：**评分表呈现的学员画像，和对话记录中观察到的实际表现，整体上是否一致？**

可能出现的情况：
- 基本一致：可以信任分数作为量化参考
- 评分表偏高：注意区分"覆盖完整"和"执行质量"
- 评分表偏低：注意评分表的扣分可能未体现学员的合理取舍
- 明显不匹配：评分表可能模板错配或残缺，分数参考价值有限

## 考官能力的五维模型

还原到"一个有经验的考官站在考生身后能观察到什么"。

| 维度 | 考官心中的问题 | 不可简化为 |
|------|--------------|-----------|
| **安全素养** | 我能放心让这个人独立接触患者吗？ | "问没问红旗征象" |
| **临床推理** | 这个人的大脑里发生了什么？从信息到决策的路径是什么样的？ | "诊断对不对" |
| **临床技能执行** | 操作是否有方法论？是临床意图驱动还是机械执行？ | "查体项目漏没漏" |
| **沟通与关系** | 患者离开诊室时是什么感受？ | "有没有说你好" |
| **专业素养** | 这个人正在成为什么样的医生？ | "穿没穿白大褂" |

### 各考核项目的观测能力矩阵

| 维度 | 病史采集 | 体格检查 | 人文沟通 | 病例分析 | 病历书写 | 精神检查 |
|------|:---:|:---:|:---:|:---:|:---:|:---:|
| 安全素养 | ★★★ | ★★★ | ★★☆ | ★★☆ | ★☆☆ | ★★★ |
| 临床推理 | ★★★ | ★★☆ | ★★☆ | ★★★ | ★★☆ | ★★★ |
| 临床技能执行 | ★★☆ | ★★★ | — | ★★☆ | ★★★ | ★★★ |
| 沟通与关系 | ★★☆ | ★★☆ | ★★★ | — | — | ★★★ |
| 专业素养 | ★☆☆ | ★★☆ | ★★★ | ★★☆ | ★★☆ | ★★★ |

> ★★★ 主要观测源  ★★☆ 辅助观测源  ★☆☆ 有限观测  — 无法观测

## 发展阶段模型

| 阶段 | 标签 | 一句话描述 | 中国教育锚点 |
|------|------|-----------|-------------|
| **S1** | 知识驱动型 | 知道该问什么，但问出来的顺序取决于教科书章节而非患者回答 | 院校教育临床前 |
| **S2** | 规则驱动型 | 能按流程完成接诊，能识别教科书级典型表现，面对变异不知所措 | 院校实习/住培第一年 |
| **S3** | 模式驱动型 | 面对典型症状能自动激活假设并定向追问，但过早收窄，忽略不一致线索 | 住培二-三年（多数人停留最久） |
| **S4** | 情境适应型 | 能根据情境灵活切换策略，直觉与分析流畅交替，在不确定中做安全决策 | 住培毕业/主治早期 |
| **S5** | 专家型 | 信息不足时做出正确判断，处理复杂不确定性，能带教他人 | 高年资主治及以上 |

**关键设计**：
1. 行为表现决定阶段，不是培训年限
2. 不同维度可以不同速——不均衡本身就是诊断信息
3. 阶段判定是概率性的，证据不足时判定区间而非精确阶段
4. 阶段的额外价值是"预期校准"——告诉老师该学员在培训进程中的相对位置

## 精神检查剖面

形式上与病史采集类似（学员问→患者答），但内核根本不同：

| | 病史采集 | 精神检查 |
|------|---------|---------|
| **目的** | 获取临床信息 | **观察与评估精神状态本身** |
| **信息来源** | 患者说的内容 | 内容 + **说的方式 + 没说出来的东西** |
| **核心操作** | 提问→倾听→追问 | 观察→互动→系统化评估11个MSE领域 |
| **沟通本质** | 工具性 | **治疗性（互动本身即是诊断和治疗）** |
| **患者状态** | 通常配合 | 可能意识障碍、思维散漫、拒绝交流 |

## 不同专业的维度解释框架

五个维度通用，但每个维度在不同专业中的观测重点和判断基准不同。专业解释框架作为**独立知识库**维护，与考官推理引擎分离：

- **考官推理引擎**：通用五维评价逻辑——从对话记录中提取证据→映射到维度→做出判断
- **专业知识库**：各专业的维度解释框架、安全标准、病例特异红旗征象清单、各阶段预期基准

新增专业只需新增知识库文件，不需修改推理引擎。

目前已定义：内科、精神科、儿科（后两者有独立解释框架，详见附录）。外科、急诊、妇产科后续补充。

## 输入物规格

### 优先级

| 优先级 | 含义 | 输入物 |
|:---:|------|--------|
| **P0** | 缺了系统无法运行 | 对话记录、病例原文 |
| **P1** | 缺了可推断但精度下降 | 考核项目类型标签 |
| **P2** | 缺了部分功能不可用 | 评分表数据（分数信度评估和评分表匹配度评估不可用，其余正常） |
| **P3** | 缺了仅影响辅助功能 | 考生专业标签、培训阶段标签、考站上下文 |
| **不可依赖** | 仅作辅助参考 | LLM情绪/意图元数据（轻量模型，偏直觉不准确，仅用于判断对话情感基调整体走向） |

### 信任层级

对所有输入物，系统持不同级别的信任：

```
对话记录  >  病例原文  >  评分表  >  考核项目类型标签  >  专业/阶段标签  >  情绪元数据
  ↑            ↑           ↑              ↑                ↑              ↑
 原始证据    高信任但    信任框架      可能标错           可能标错        不可信任
 (最高信任)  可能不完整  不信任结论    (应从内容推断)     (应从内容推断)   (轻量模型)
```

**输入物不能盲信**：
- 对话记录本身是原始证据，即使有截断，已有部分是真实的。但如果明显不完整（如有头无尾、缺少关键环节），系统应在数据局限性声明中标注
- 病例原文可能缺少显式标注的红旗征象。系统需自行从病例中提取，而非依赖外部标签
- 评分表可能模板错配、残缺、与病例不匹配。系统不依赖其结论，以对话记录为最终判断依据
- 类型标签和专业标签可能标错。系统应从对话内容中交叉验证（如对话中包含查体操作→必然不是纯病史采集）

### 数据结构

```ts
interface AssessmentInput {
  // 考生信息（P3）
  candidate?: {
    name?: string
    specialty?: string
    phase?: string
  }

  // 病例信息（P0）
  case_info: {
```

## 分析流程架构

一个有经验的考官不会同时做所有事。他经历一个自然的认知递进。将这个认知过程结构化为三阶段流水线：

### 三阶段流水线

```
输入: 对话记录 + 病例原文 + (评分表) + (专业维度框架)
  │
  ├── 剖面A分析 ──┐
  ├── 剖面B分析 ──┤  ← 阶段一：多剖面独立分析（可并行）
  ├── 剖面C分析 ──┘     每剖面五层深度分析
  │       │              产出：结构化剖面报告
  │       ▼
  │   跨剖面整合分析       ← 阶段二（依赖阶段一全部产出）
  │       │              维度综合判定 + 根因分析 + 数据局限性
  │       │              产出：整合分析报告
  │       ▼
  │   四Part报告生成       ← 阶段三（将分析产物翻译为叙事）
  │       │              
  ▼       ▼
输出: ComprehensiveEvaluation
```

### 阶段一：多剖面独立分析

**目标**：对每个考核项目，生成独立的结构化剖面报告。

**输入**：该项目的对话记录 + 病例原文 + 专业维度框架

**分析行为**（五层递进）：

1. **L1 信息覆盖**：对话记录 vs 病例原文，逐项对照关键信息点。产出覆盖热力图和遗漏清单
2. **L2 行为策略**：识别策略模式（假设驱动/模板覆盖/随机跳跃），分析问题类型分布、序列逻辑、时间分配
3. **L3 认知过程**：从提问序列反推假设形成-验证-收窄过程，判定推理模式（直觉型/分析型/混合），检测过早关闭、确认偏误
4. **L4 安全行为**：对照病例红旗征象清单，匹配对话中的筛查行为，分析安全行为与假设活动的时间关联
5. **L5 关系质量**：识别情感线索、评估回应质量、描绘关系弧线

五个层次的分析结果 + 层次之间的关联发现 + 跨剖面线索标记 → 一份结构化剖面报告。

**可并行**：如果有多个考核项目，各剖面独立运行，互不依赖。

### 阶段二：跨剖面整合

**目标**：收集所有剖面报告，进行跨维度模式识别和根因分析。

**输入**：所有剖面报告（阶段一产出）+ 临床认知偏误知识（知识库）

**分析行为**：

1. **维度证据聚合**：对每个维度，从所有剖面收集证据片段，判定收敛/发散/不足
2. **不均衡检测**：五维之间的发展阶段差异，识别"矮板"
3. **根因搜索**：跨维度的重复模式 → 提出根因假设 → 排除替代解释 → 确认根因
4. **数据局限性判定**：对照观测矩阵，分类声明可评估/部分可评估/不可评估

**上下文管理**：阶段二不接触原始对话。输入是阶段一的结构化报告，体积远小于原始数据。

### 阶段三：报告生成

**目标**：将阶段一和阶段二的分析产物翻译为面向读者的四Part报告。

**输入**：剖面报告（阶段一）+ 整合分析（阶段二）+ 评分表整体匹配度

**核心任务**：分析与表达分离。同一个分析结果可以有不同的叙事方式（学员版/教师版/摘要版）。冷冰冰的结构化发现 → 有温度的叙事。

---

## 报告边界

- **基础单元**：一站一报告，覆盖该站内所有考核项目
- **跨站综合分析**：独立功能，收集多站报告后做跨站聚合，不在本设计范围内
- 一站内的多个考核项目天然属于同一临床过程，逻辑上是一体的

### 精神科

**本质差异**：面谈本身即是诊断工具。MSE在对话中同步完成。

| **沟通与关系** | **三角沟通质量**：家长报告与儿童参与之间的平衡 |
| **专业素养** | **儿童保护伦理**：何时突破家长隐私？家长意愿 vs 儿童最佳利益 |

## 附录A：评分表病例锚定解析设计

### A.1 设计约束

系统的核心约束是：**评分表由客户自行上传和绑定，真人考官和AI考官基于同一张评分表评分**。这决定了评分表的权威性始终在客户配置的原表上，系统的任何解析都只是"展开辅助"而非"替换"。

### A.2 当前评分表体系问题

经2026-06-24审查，发现以下问题：

| 问题 | 表现 |
|------|------|
| **三套模板系统不互通** | `packages/shared/data/score-tables/index.js`（共享注册）、`apps/admin/src/data/templates/`（23个JSON）、`score-sheet-v1.json` + `score-sheet-generator.js`（唯一实际生成路径）各自独立 |
| **病例特异性为零** | 生成器只做 `{symptom}` 占位符替换，一个"乏力待查"和一个"腹痛待查"的评分表几乎完全一样 |
| **IM专业模板被浪费** | 16个内科浙江省方案模板存在但从未被任何病例绑定（全部使用TPL-STD） |
| **评分规则是存在性判断** | "问了→得分，没问→没分"，无法区分问了1个问题还是5个追问 |
| **查体评分点靠正则** | 22条正则从病例文本匹配，查体文本中未出现的阴性体征不进入评分表 |

### A.3 问题一：未解析的情况下，真人和AI能否评分？

**结论：都能评分。一致性取决于评分表颗粒度，根源问题不在AI vs 真人，而在评分表本身的模糊性。**

#### 真人评分路径

真人考官拿一张通用评分表，看到"起病情况与诱因 (8分)"这样的条目，脑子里自动完成：

1. **临床映射**：这个胸痛病例 → 起病要问突发还是渐进、有无劳力诱发、有无情绪应激
2. **对话比对**：学员问了哪些？遗漏了哪些？

真人能做到，依赖的是自身的临床知识补全了评分表的模糊地带。

#### AI评分路径

AI拿到同样的评分表 + 对话记录 + 病例原文，能做同样的映射。LLM有足够的临床知识完成"起病情况与诱因"到具体追问点的展开，然后逐项比对对话记录。

#### 一致性分析

一致性差的根因在**通用评分表自身的模糊性**。同一个条目"起病情况与诱因 (8分)"：
- 考官A认为"问了起病时间和方式就给满分"
- 考官B认为"没问具体的劳力关系，扣2分"

**这个问题在两个真人考官之间同样存在**——OSCE考试中，评分表越通用，考官间信度越低。对培训场景而言可接受，对高利害考核需要解析辅助。

```
通用评分表一致性规律：

评分表颗粒度粗 ──→ AI vs 真人一致性 ≈ 真人 vs 真人一致性
        ↓
评分表颗粒度细 ──→ 各方一致性显著提升
（有sub_items展开）
```

### A.4 问题二：解析应该放在哪个阶段？

**结论：解析不能放在病例生成时，应分别放在考试配置阶段（考试模式）和训练会话初始化阶段（训练模式），两处调用同一套解析逻辑。**

#### 为什么不能在病例生成时解析

病例生成时只知道病例自身信息，不知道将来会挂哪张评分表、用于哪个专业、配置在哪个考站。评分表是客户后来选择和绑定的，时间上晚于病例生成。

#### 考试模式：解析发生在考试配置阶段

管理员创建考试配置时，病例 + 评分表 + 考站方案 + 专业四者第一次同时已知：

```
病例（已有）  +  评分表模板（客户选择）  +  考站配置（含项目分值映射）
                        │
                        ▼
               【考试配置阶段】← 解析触发
                        │
              解析函数：parseScoreSheet()
              输入：病例basic + 评分表模板 + 考站项目映射 + 专业
              输出：解析后的评分表（原条目保留，sub_items展开）
                        │
                        ▼
           ┌────────────┼────────────┐
           ▼                         ▼
     真人考官评分              AI考官评分
  （同一张解析后评分表）    （同一张解析后评分表）
```

#### 训练模式：解析发生在会话初始化阶段

训练没有"管理员配置考试"这一步，但在**用户选完病例、进入考站的那一刻**，系统同样已获知全部必要信息：

```
训练端流程：

用户选病例 → 进入考站 → [解析触发] → 评分表锚定 → AI评分使用
                              ↑
                    此时已知：病例basic + score_sheet_template + 考站方案 + 专业
                    与考试配置阶段的信息集完全相同
```

#### 解析逻辑共享

解析函数不关心调用入口是谁，输入输出一致：

```
┌──────────────────────────────────────────────────┐
│              parseScoreSheet()                      │
│                                                      │
│  输入:                                                │
│    - basicData      病例基本信息                      │
│    - templateCode   评分表模板编码                     │
│    - specialty      专业名称                          │
│    - stationScheme  考站配置（可选，含项目分值映射）    │
│                                                      │
│  输出:                                                │
│    - 与原表结构一致的评分表，每个条目追加 sub_items      │
│    - 原始条目和分值不变（客户权威保留）                 │
│                                                      │
│  调用方:                                              │
│    - 考试模式: ExamCreate.vue 提交配置时              │
│    - 训练模式: 训练端 session 初始化时                 │
│    - 成绩报告: Part B L1 覆盖分析时（读取已解析结果）   │
└──────────────────────────────────────────────────┘
```

### A.5 解析后的数据格式

解析不替换客户上传的原条目，而是在其下追加 `sub_items`：

```json
// 原始（客户上传/选择的）
{ "item": "起病情况与诱因", "score": 8 }

// 解析后（系统追加）
{
  "item": "起病情况与诱因",
  "score": 8,
  "sub_items": [
    { "point": "询问起病时间及起病方式（突发/渐进）", "score": 2 },
    { "point": "询问有无劳力诱发因素", "score": 2 },
    { "point": "询问有无情绪激动/应激诱发", "score": 2 },
    { "point": "询问起病时症状严重程度", "score": 2 }
  ]
}
```

设计要点：
- **原条目不修改**：`item` 和 `score` 保持客户配置的权威值
- **sub_items 总分为原条目分值**：展开不改变总分结构
- **sub_items 是打分参考**：真人和AI可将它作为细粒度评分指南，但最终分数仍然落在原条目上
- **解析结果存入考试实例/训练会话**：解析只发生一次，产物被双方共用

### A.6 解析内容来源

从病例原文中提取病例锚定信息，按条目类型匹配展开规则：

| 条目类型 | 展开策略 | 示例 |
|---------|---------|------|
| **主诉追问类** | 从 chief_complaint + present_illness 提取症状维度 | "胸痛" → 部位/性质/放射/加重缓解/伴随 |
| **伴随症状类** | 从 diagnosis.differential 反推鉴别所需阴性症状 | 鉴别含"心梗" → 追问心悸/出汗/恶心 |
| **红旗筛查类** | 从 full_text 提取高危关键词匹配红旗模板 | "黑便/呕血/血色素低" → 内出血筛查项 |
| **风险因素类** | 从 past_history + diagnosis 匹配疾病特定风险因素 | "肝硬化" → 饮酒/乙肝/药物/输血史 |
| **查体类** | 从 physical_exam 提取阳性体征 + 保底关键检查项 | 腹部查体 → 视触叩听四项展开 |
| **通用固定项** | 既往史/个人史/家族史/系统回顾保持原样 | 不需要展开 |
